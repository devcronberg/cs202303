    Introdution til .NET

    Om modulet

    Dette modul vil introducere dig til programmering. Hvis du kender til programmeringssprog mv kan du bare springe det over. Herefter er der en intro til .NET Framework/.NET Core. Sluttelig skal du tage stilling til hvilket udviklingsmiljø du vil benytte.

    Formål med programmering

    Video Formål med programmering (9 min.)

    Se video

    Det overordnede formål med programmering er jo, at skabe instruktioner til en CPU (processor). Der er mange forskellige processorer, og mange forskellige arkitekturer (X86, X64, ARM med videre)

    En processor består i bund og grund af en stor samling transistorer (elektrisk kontakt), som kombineret i såkaldte gates (digitale porte) skaber mulighed for at beregne og opbevare data i en rasende fart.

    En processor har et instruktionssæt som - i bund og grund - er binære værdier der åbner/lukker/tænder/slukker for kontrollinjer til de forskellige komponenter i processoren (registrer, beregningsenhed, I/O med videre).

    Hvis du har lyst kan du se et eksempel på en SAP-CPU (simple as possible CPU) som jeg bruger i andre kurser.

    Her er der et begrænset antal kontrollinjer som gør det muligt at skrive programmer til “processoren”. Så binærer instruktioner som:

    1010 0111 0
0000 0101 0
1000 0000 1
0110 0000 0

    betyder i virkeligheden:

    Hent til tal fra en adresse i RAM
Læg det sammen med sig selv
Placer det i RAM igen

    Men i stedet for at påvirke kontrollinjer direkte med binære koder, er det noget nemmere at få hjælp fra en “Assembler”, som egentlig blot oversætter “bogstavkoder” til binære værdier (samt tilføjer features som blandt variabler):

    LIA 3
STA F
ADD F
OUTA
JMP 3

    Koden beregner en tre tabel.

    Se eventuelt mere på devcronberg/sap-cpu - men det er ikke vigtigt for din videre færd ind i C#.

    Maskinkode / binære instruktioner

    • her 16 bit instruktioner:
    1100010001000011 0011101001110010
1000110001110000 1111101110000001
...
    • der også kan skrives som (hex)
    C443 3A72
8C70 FB81
...

    Maskinsprog / Assembler

    De binære instruktioner er hvad alt kode i sidste ende bliver til, men det er lidt nemmere at skrive Maskinsprog/Assembler (lavniveau)

    • Meget tæt på CPU instruktioner
    • Variabler
    • Flow
      • Betingelser
      • Jump
    main:
    mov ax,'00'
    mov di,counter
    mov cx,digits+cntDigits/2
    cld
    rep stosw
    inc ax
    mov [num1 + digits - 1],al
    mov [num2 + digits - 1],al
    mov [counter + cntDigits - 1],al
    jmp .bottom

    Højniveau sprog

    Og nemmere at benytte et højniveau sprog (her C#):

    // Her er en kommentar
int antal = 3;
bool skriv = true;
string navn = "mikkel";
if (antal < 5)
{
    for (int i = 0; i < antal; i++)
    {
        if (skriv)
        {
            Console.WriteLine(navn);
        }
    }
}
else
{
    Console.WriteLine("Antal er for stort");
}

    Oversættelse af kode

    Video Diverse programmeringssprog (6 min.)

    Se video

    Traditionelt oversættes kildekode direkte til eksekverbar kode

    .NET sprog oversættes til IL-kode, som efterfølgende af en runtime oversættes til eksekverbar kode

    Fra C#.NET til IL til ASM

    Her er et kort eksempel på kompilering fra C# til IL til ASM (assembler) (skabt ved hjælp af https://sharplab.io/)

    Her er C#-kode:

    using System;

namespace Demo
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Program start");
            // Tæller fra 1-5
            for (int i = 1; i < 6; i++)
            {
                VisTal(i);
            }
            Console.WriteLine("Program slut");
        }

        private static void VisTal(int i)
        {
            Console.WriteLine(i);
        }
    }
}

    Som konverteres til IL-kode:

    .class private auto ansi '<Module>'
{
} // end of class <Module>

.class private auto ansi beforefieldinit Demo.Program
    extends [mscorlib]System.Object
{
    // Methods
    .method private hidebysig static
        void Main (
            string[] args
        ) cil managed
    {
        // Method begins at RVA 0x2050
        // Code size 49 (0x31)
        .maxstack 2
        .locals init (
            [0] int32,
            [1] bool
        )

        IL_0000: nop
        IL_0001: ldstr "Program start"
        IL_0006: call void [mscorlib]System.Console::WriteLine(string)
        IL_000b: nop
        IL_000c: ldc.i4.1
        IL_000d: stloc.0
        // sequence point: hidden
        IL_000e: br.s IL_001d
        // loop start (head: IL_001d)
            IL_0010: nop
            IL_0011: ldloc.0
            IL_0012: call void Demo.Program::VisTal(int32)
            IL_0017: nop
            IL_0018: nop
            IL_0019: ldloc.0
            IL_001a: ldc.i4.1
            IL_001b: add
            IL_001c: stloc.0

            IL_001d: ldloc.0
            IL_001e: ldc.i4.6
            IL_001f: clt
            IL_0021: stloc.1
            // sequence point: hidden
            IL_0022: ldloc.1
            IL_0023: brtrue.s IL_0010
        // end loop

        IL_0025: ldstr "Program slut"
        IL_002a: call void [mscorlib]System.Console::WriteLine(string)
        IL_002f: nop
        IL_0030: ret
    } // end of method Program::Main

    .method private hidebysig static
        void VisTal (
            int32 i
        ) cil managed
    {
        // Method begins at RVA 0x208d
        // Code size 9 (0x9)
        .maxstack 8

        IL_0000: nop
        IL_0001: ldarg.0
        IL_0002: call void [mscorlib]System.Console::WriteLine(int32)
        IL_0007: nop
        IL_0008: ret
    } // end of method Program::VisTal

    .method public hidebysig specialname rtspecialname
        instance void .ctor () cil managed
    {
        // Method begins at RVA 0x2097
        // Code size 8 (0x8)
        .maxstack 8

        IL_0000: ldarg.0
        IL_0001: call instance void [mscorlib]System.Object::.ctor()
        IL_0006: nop
        IL_0007: ret
    } // end of method Program::.ctor

} // end of class Demo.Program

    Runtime

    Video Runtime (6 min.)

    Se video

    En runtime i .NET skal du se som en nødvendighed for at kunne afvikle IL-kode. Før 2020 fandtes der er en del forskellige runtime (.NET Framework, .NET Core, Mono og Xamarin), men i november 2020 blev .NET 5 frigivet (sammen med C# 9), og den er første skridt til at samle de gamle runtimes til en samlet runtime.

    Download de forskellige runtime’s

    Se .NET som en virtuel maskine med en generisk samling af komponenter man kan skrive kode til, og som nok selv skal sørge for at applikationen bliver afviklet på den pågældende platform.

    Internt består en .NET runtime af komponenter som:

    • CLR, som sørger for afvikling
    • BCL, som er en stor samling klassebiblioteker der stiller funktionalitet til rådighed
    • Komponenter brugt i forbindelse med udvikling (debuggere mv)
    • Komponenter brugt i forbindelse med sikkerhed (signering af kode mv)

    BCL

    Video BCL (5 min.)

    Se video

    Især BCL’en er vigtig for C# udviklere, for her findes alle de klasser og strukturer som MS stiller til rådighed for at hjælpe med udvikling af applikationer.

    Komponenterne er placeret i namespace’s for at skabe en logisk struktur og indkapsling, og samtidigt undgå navnesammenfald.

    De fleste klasser og strukturer kan tilgås under “System”, og ved hjælp af punktumnotation kan BCL’en “browses/bladres”.

    En stor del af det at lære at kode i C# handler om, at lære hvordan man nemmest

    • åbner en fil
    • taler med en database
    • skaber et http-kald
    • finde noder i en xml-fil
    • finder et tilfældig tal
    • Kryptere en tekst
    • osv…

    og typerne til at håndtere det finder i selve BCL’en.

    NuGet

    Video NuGet (7 min.)

    Se video

    NuGet er Microsofts store pakkebibliotek hvor der findes yderligere generisk kode. Pakker er tilføjet af både Microsoft og af 3. parts firmaer, grupper eller enkelt personer. I starten af 2021 var der over 230.000 pakker tilgængelige, og de kan tilføjes ed download, gennem konsol eller (nemmest) gennem Visual Studio eller Visual Studio Code. Mange af pakkerne er open source-projekter fra GitHub.

    Hvad kan man udvikle med .NET (og C#)

    Hvis først du kender til et .NET sprog (C#, F#, VB.NET) har du mulighed for at skabe mange forskellige typer af applikationer.

    Konsol

    Dette kursus tager udgangspunkt i en konsolapplikation for at kunne holde fokus på teorien og syntaksen. Det er den mest simple type af applikation, men bliver faktisk benyttet en del til værktøjer og administrative applikationer. Og som en af de få applikationstyper (bortset fra webapplikationer) kan den afvikles på alle supporterede platforme.

    WinForm

    Hvis du ønsker at skabe en desktop applikation har du forskellige muligheder. Du kan vælge at udvikle en traditionel Windows Forms-applikation, som er bundet til interne komponenter i Windows. Denne typer applikation har levet i mange år, og er opgraderet til forskellige runtimes. I skrivende stund kan den findes på listen over mulige projekter i Visual Studio som ”Windows Forms (.NET)”.

    Med Windows Forms kan du meget hurtigt skabe en brugerflade med alle mulige former for kontroller, og så binde det hele sammen med C# kode.

    WPF

    Du kan også vælge en WPF (Windows Presentation Foundation) applikation som er noget nyere end Windows Forms, og ikke helt så bundet til interne komponenter i Windows. Der er nogen flere muligheder i en WPF applikation, men den er fortsat bundet til Windows-platformen.

    I skrivende stund kan den findes på listen over mulige projekter i Visual Studio som ”WPF App (.NET)”.

    I skrivende stund er Microsoft ved at færdiggøre et projekt kaldet WinUI , og det vil sikkert blive den anbefalede måde (måske fra .NET 6) at skabe desktop applikationer på.

    MAUI

    MAUI er Microsofts nyeste bud på et cross platform UI framework. Med MAUI kan du bygge både desktop og mobile applikationer. Det er en videreudvikling af Xamarin Forms, og nye projekter bør måske kigge på MAUI fremfor Xamarin.

    Mobil

    Du kan også skabe forskellige typer af mobile applikationer til Android og iOS. Microsoft købte for nogle år siden et firma der hed Xamarin som havde skabt et framework til at udvikle mobile applikationer. Navnet Xamarin fremgår stadig af projekttyperne i Visual Studio, og det er i skrivende stund måden at skabe mobile applikationerne på med C#.

    Web

    Hvis du ønsker at skabe web applikationer har Microsoft tre forskellige projekttyper du kan vælge i mellem. Det mest solide og komplette framework hedder ASP.NET MVC (står for Model View Controller), og det benyttes til de traditionelle serverbaserede webapplikationer. Alternativet er det mindre, men også meget nemmere, framework kaldet ASP.NET Web Pages. Hvis du blot skal skabe et par websider, er det uden tvivl det nemmere at komme i gang med. Som noget nyt kan du også vælge at skabe SPA applikationer (Single Page Applications) med et nyt framework kaldet Blazor . Det benytter helt nye standardiseret måder at afvikle IL kode (og dermed C#) direkte i browseren. I skrivende stund er dette framework meget nyt, men den teknologi det er bygget på (WebAssembly forkortet WASM) er på vej frem i webudvikling, og det bliver spændende at se om Blazor bider sig fast.

    Service/API

    Hvis du ønsker at skabe en servicebaseret applikation som kan være kernen i samling af forskellige applikationer, bør du se nærmere på gRPC . Det er oprindeligt Google’s framework til RPC (Remote Procedure Call), og er nu tæt på industristandard for den typer af applikationer på mange platforme og i mange programmeringssprog. Projekttyper relateret til gRPC er i skrivende netop tilføjet .NET (og Visual Studio).

    Database

    Slutteligt kan du næsten ikke skabe en applikation uden at involvere en database. Der findes masser af muligheder for at tale med alle mulige former for databaser i .NET, men de fleste vælger at benytte et ORM produkt (Object–Relational Mapping). Det giver mulighed for at abstrahere meget at databasekode væk, og i stedet fokusere på forretningslogik. Microsoft eget ORM produkt hedder Entity Framework (EF), og kan spare en frygtelig masse tid ved udvikling af en applikation.

    EF er en integreret del .NET og udviklingsmiljøerne.
